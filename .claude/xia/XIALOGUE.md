Module 33-best-practice-patterns (2026-03-25) borrows three non-obvious patterns from claude-code-best-practice: (1) CLAUDE.md monorepo loading — ancestor loading (upward, immediate at startup) vs descendant loading (downward, lazy when files read), siblings never load; global `~/.claude/CLAUDE.md` always loads; (2) Cross-model review workflow — Plan (Claude Opus) → QA review (second model, additive-only: inserts "Phase 2.5" findings, never rewrites) → Implement (Claude Opus, new session) → Verify (second model); (3) Three underdocumented agent frontmatter fields: `isolation: "worktree"` (auto-cleaned temporary worktree per agent), `background: true` (non-blocking), `color` (CLI output color for multi-agent distinction).

Module 32-cc-sdd-patterns (2026-03-25) borrows four spec quality patterns from cc-sdd: (1) EARS format — five structured acceptance criteria patterns (WHEN/THEN, WHILE/SHALL, IF/THEN, WHERE/SHALL, THE/SHALL) replacing plain checkboxes with independently testable statements; (2) Steering files — three persistent project memory files (product.md, tech.md, structure.md) in `.claude/steering/` loaded by every planning command, bootstrapped from codebase analysis, updated additively; (3) Feature type classification — four types (Greenfield/Extension/Simple addition/Complex integration) calibrate design discovery depth before planning; (4) `(P)` parallel task markers — annotation convention for tasks.md indicating concurrent-safe tasks, with strict eligibility rules (no data dep, no shared state, no prerequisite review).

Module 31-gsd-planning-patterns (2026-03-25) borrows four planning conventions from get-shit-done: (1) `.planning/` directory convention — PROJECT.md (core value + out-of-scope) + REQUIREMENTS.md + ROADMAP.md + STATE.md + phases/; (2) STATE.md living project memory (≤100 lines) — current position (phase X of Y, progress bar), velocity metrics, accumulated decisions/blockers, session continuity — read first every session, updated after every significant action; (3) REQUIREMENTS.md template with v1/v2 buckets, explicit out-of-scope table with reasons, and requirement→phase traceability matrix; (4) `/forensics` command — read-only post-mortem that analyzes git commit frequency (stuck-loop = same file in 3+ consecutive commits), missing planning artifacts (no SUMMARY.md = phase not closed), orphaned worktrees, and time-gap abandonment signals.

Module 30-plan-lifecycle (2026-03-25) borrows the 4-command plan pipeline from humanlayer/humanlayer: (1) `create_plan` — skeptical, interactive plan creation with parallel research agents, design option presentation, phasing approval, and a structured plan template with explicit automated + manual success criteria and "What We're NOT Doing" section; (2) `iterate_plan` — surgical plan updates grounded in codebase research, confirm-before-change, maintains automated/manual split; (3) `implement_plan` — phase-by-phase execution tracking checkboxes in the plan file, stops on reality/plan mismatch, pauses after automated checks for human to confirm manual steps; (4) `validate_plan` — post-implementation audit running all automated criteria, generating ✓/⚠️/✗ report per phase, listing remaining manual items. Plans stored in `.claude/plans/YYYY-MM-DD-description.md`.
